All write flows are direct: read the calendar, submit a write, handle
rejection. No intents, no quotes, no shopping cart, no confirmation step.

## Reserve a batch of intervals

One call creates blocks across one or more 15-minute intervals. Atomic across
the whole request — either everything is booked or nothing is.

```bash curl
curl -X POST https://app.opencomputer.dev/api/capacity/reservations \
  -H "X-API-Key: $OPENCOMPUTER_API_KEY" \
  -H "Idempotency-Key: nightly-agents-2026-04-25" \
  -H "Content-Type: application/json" \
  -d '{
    "intervals": [
      { "startsAt": "2026-04-25T09:00:00Z", "endsAt": "2026-04-25T09:15:00Z", "capacityGb": 16 },
      { "startsAt": "2026-04-25T09:15:00Z", "endsAt": "2026-04-25T09:30:00Z", "capacityGb": 16 }
    ]
  }'
```

Response (success):

```json
{
  "status": "applied",
  "reservedBlocks": 32,
  "cancellableBlocks": 24,
  "nonCancellableBlocks": 8,
  "blocks": [
    {
      "blockId": "9f67b8f7-7b91-4d2d-b1cb-19d0d0a14562",
      "startsAt": "2026-04-25T09:00:00Z",
      "endsAt":   "2026-04-25T09:15:00Z",
      "isCancellable": false,
      "cancellableUntil": null
    }
    // ... one entry per block
  ]
}
```

### Validation rules

| Rule | Behavior |
| --- | --- |
| `startsAt` and `endsAt` are UTC and 15-minute aligned | Reject misaligned intervals. |
| `endsAt == startsAt + 15 minutes` | Reject multi-interval spans; enumerate them instead. |
| `capacityGb` is a positive integer | Reject zero, negative, or fractional. |
| `startsAt >= earliestReservableStart` | Reject intervals that are too soon to reserve. |
| Requested capacity fits current `reservableGb` | Reject instead of overbooking. |
| `Idempotency-Key` is stable for retries | Replaying returns the original result. |

### Rejected write

```json
{
  "error": "capacity_not_available",
  "intervals": [
    {
      "startsAt": "2026-04-25T09:00:00Z",
      "requestedGb": 80,
      "reservableGb": 30,
      "reason": "insufficient_capacity"
    },
    {
      "startsAt": "2026-04-25T09:15:00Z",
      "requestedGb": 80,
      "reservableGb": 0,
      "reason": "too_soon"
    }
  ]
}
```

The response lists every problematic interval, so the client can retry with
adjusted numbers in a single round-trip.

Common rejection reasons:

| `reason` | Meaning |
| --- | --- |
| `insufficient_capacity` | Request exceeds current `reservableGb`. |
| `too_soon` | `startsAt` is before `earliestReservableStart`. |
| `misaligned` | `startsAt`/`endsAt` not on the 15-minute grid. |
| `reservation_limit_exceeded` | Exceeds the org's `reservationLimitGb` for the interval. |

## Reserve exactly one block with a caller-chosen ID

For single-block reservations where you want idempotency keyed to a UUID you
generate client-side:

```bash curl
curl -X PUT \
  https://app.opencomputer.dev/api/capacity/blocks/3a94b5e6-2e4f-4a2b-b58a-53c41d61d9bd \
  -H "X-API-Key: $OPENCOMPUTER_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "startsAt": "2026-04-25T09:00:00Z",
    "endsAt":   "2026-04-25T09:15:00Z"
  }'
```

This is equivalent to reserving `capacityGb: 1` for that interval. Because
you chose the ID, replaying the same `PUT` with the same body is naturally
idempotent — the same key always names the same block.

| Case | Behavior |
| --- | --- |
| No block exists for `(org, blockId)` | Validate capacity and create the block. |
| Existing live block with same `startsAt`/`endsAt` | Return the existing block; no new GB reserved. |
| Existing live block with different `startsAt`/`endsAt` | Reject `block_id_conflict`. |
| Block was cancelled or settled | Reject `block_id_not_reusable`. IDs are never recycled. |
| Same UUID exists in another org | Ignored. Block identity is scoped by org — same UUID in two orgs means two different blocks. |

## Idempotency keys

Both `POST /reservations` and `POST /cancellations` accept an
`Idempotency-Key` header. The server stores the key with the request body
hash and the response. Retries with the same key and body return the cached
response — even if the original write succeeded, the retry is safe.

| Behavior | Response |
| --- | --- |
| Same key, same body | Cached response from first call. |
| Same key, different body | `idempotency_key_conflict`. |
| No key | Request is processed normally; retries are the client's problem. |

Always send a stable `Idempotency-Key` for reservation and cancellation
writes. Build it from a deterministic name for the operation, e.g.
`reserve-nightly-agents-2026-04-25`, so your retry loop sends the same key.

## Cancel blocks by ID

Cancellation targets specific block IDs. It never targets a time range or a
GB amount. Atomic all-or-nothing: if any block in the request is not
cancellable, none are cancelled.

```bash curl
curl -X POST https://app.opencomputer.dev/api/capacity/cancellations \
  -H "X-API-Key: $OPENCOMPUTER_API_KEY" \
  -H "Idempotency-Key: cancel-nightly-2026-04-25" \
  -H "Content-Type: application/json" \
  -d '{
    "blockIds": [
      "9f67b8f7-7b91-4d2d-b1cb-19d0d0a14562",
      "3a94b5e6-2e4f-4a2b-b58a-53c41d61d9bd"
    ]
  }'
```

Response (success):

```json
{
  "status": "cancelled",
  "cancelledBlocks": 2,
  "cancelledGb": 2,
  "cancelledGbMinutes": 30,
  "blocks": [
    { "blockId": "9f67b8f7-7b91-4d2d-b1cb-19d0d0a14562", "startsAt": "2026-04-25T09:00:00Z", "endsAt": "2026-04-25T09:15:00Z" },
    { "blockId": "3a94b5e6-2e4f-4a2b-b58a-53c41d61d9bd", "startsAt": "2026-04-25T09:15:00Z", "endsAt": "2026-04-25T09:30:00Z" }
  ]
}
```

Response (rejection):

```json
{
  "error": "blocks_not_cancellable",
  "blocks": [
    { "blockId": "9f67b8f7-7b91-4d2d-b1cb-19d0d0a14562", "reason": "not_cancellable" },
    { "blockId": "3a94b5e6-2e4f-4a2b-b58a-53c41d61d9bd", "reason": "already_started" }
  ]
}
```

Common rejection reasons:

| `reason` | Meaning |
| --- | --- |
| `not_cancellable` | `cancellableUntil` has already passed. |
| `already_started` | The interval has begun; reservation is settled. |
| `already_cancelled` | A previous cancel retired this block. |
| `not_found` | Block does not exist, or belongs to another org. |

## Cancelling a single block

For one-block cancellation, `DELETE` is a shortcut:

```bash curl
curl -X DELETE \
  https://app.opencomputer.dev/api/capacity/blocks/9f67b8f7-7b91-4d2d-b1cb-19d0d0a14562 \
  -H "X-API-Key: $OPENCOMPUTER_API_KEY"
```

`204 No Content` on success. On failure, the response body matches the
`blocks_not_cancellable` shape with a single entry.

## Retry pattern: "cancel everything cancellable"

The atomic all-or-nothing rule makes "cancel everything that's still
cancellable" a deliberate loop, not a single call:

```bash
# 1. List currently cancellable blocks in the window
curl "https://app.opencomputer.dev/api/capacity/blocks?from=...&to=...&isCancellable=true" \
  -H "X-API-Key: $OPENCOMPUTER_API_KEY"

# 2. POST those IDs to /cancellations

# 3. If the server returns blocks_not_cancellable, refresh the list and retry
#    with the remaining IDs.
```

Blocks race into non-cancellable state as their `cancellableUntil` passes.
Between the list and the cancel call, a block may mature. The rejection
response names every blocking ID, so refresh is cheap.

V1 does not accept a `skipIneligible` flag. Strict all-or-nothing is the
only mode — it keeps retries explicit and cancellation behavior predictable.

## Concurrency and atomicity summary

| Operation | Atomic across | On failure |
| --- | --- | --- |
| `POST /reservations` | All intervals in the request | No blocks created. |
| `POST /cancellations` | All blocks in the request | No blocks cancelled. |
| `PUT /blocks/{id}` | One block | No block created. |
| `DELETE /blocks/{id}` | One block | No block cancelled. |

Parallel writes to the same interval from the same org serialize server-side.
Whichever write commits first wins the remaining capacity; later writes see
the updated `reservableGb` in their validation error.
